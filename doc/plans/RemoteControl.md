# Plan for lava-cli remote control

Motivation: Make debugging easier with a new command-line tool to break and inspect
running trace replays.

## Functionality

The basic functionality is a new option to `lava-replay` that makes it daemonize itself:

```
lava-replay --daemonize <my trace file>.vk
```

When called like this, it launches itself as a background process which then waits for
further instructions before starting to run. The foreground process finds the TCP port to
bind to, writes it and a default thread index to `~/.lavatube/config.json` and returns
immediately.

These instructions come from a new tool we can call `lava-cli`.

The basic instructions are - in prioritized order:
* `lava-cli thread N` - updates the stored current thread index
* `lava-cli status` - show the status of our connections
* `lava-cli continue` - continues the replay until the end, receiving debug info from all
   threads, and if replay fails then print the last debug information received
* `lava-cli step [frame|call|calls X|frames X]` - step the given number of calls or frames ahead
   in the current thread, then pause again
* `lava-cli stop` - stops the replay
* `lava-cli info` - show input parameters and important state
* `lava-cli tell <parameter name>` - print given input parameter (nested if a chain of structs)
* `lava-cli show <object type> [index|id] <number>` - print given globally tracked object and its metadata
* `lava-cli list <object type>` - list all objects of given type tracked globally and their status
* `lava-cli goto <command...>` - continue until we encounter the first command with given name on
  the current thread (eventually from the given comma-separated list and match regex), fail and inform
  if not matching any known Vulkan command
* `lava-cli save-image <image index> <filename>` - write contents of image given by index to the given filename
* `lava-cli save-buffer <buffer index> <filename>` - write contents of buffer given by index to the given filename
* `lava-cli debug <level>` - change global debug level

## How

We can implement this by using our callbacks. If we are in daemon mode, once we have determined which
API call is next, we can attach a lava-cli callback to that pauses the replay once run, unless
some counting sentinels are higher than zero. If `goto` is used, we can attach a callback to the
specified command(s) instead and set counting sentinels to practical infinity until we hit our
target command.

Note that we deliberately pause _after_ command execution so that we can inspect the results from
the command.

We need to autogenerate the sentinel callback functions for every command.

We also need to autogenerate print and info functions for each command and struct type.

## Multi-threading

Lavatube is multithreaded and traces _can_ be heavily multi-threaded. If we pause one thread,
the other threads will continue until they hit their next synchronization point. We will want
to make sure we let all threads hit their next synchronization point before we return control
to `lava-cli` (possibly means that we need an atomic state machine status in each running
thread to say whether we are running or waiting, unless there is some other way to check this,
eg see if we are at the next synchronization point or not).

## Security

We must allocate all our TCP bindings before we enter sandbox level higher than 1. Sandbox
should still work with daemonization, but must be done in the right order so we create
resources needing rights before we enter a sandbox level that prohibits them.

## Implementation steps

1. First step is to to implement the sentinel callback(s). Possibly autogenerated. Once we hit
   the designated pause point, for now we can just log success (and print some info about the
   state of our threads), and then continue.
2. Second step is to implement the daemonization, `lava-cli` with `continue` and stop continuing
   automatically but wait for this command.
3. Third step is to implement the `step` and `stop`.
4. Fourth is to add the autogeneration for `info`.
